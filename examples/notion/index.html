Evidence-first design grammar

Trace how a site is built before you try to remake it.

Style Trace is an MCP server that opens public websites, inspects layout and interface signals, and returns a compact design grammar you can actually build against. It favors evidence over taste, structure over moodboards, and reviewable output over vague inspiration.

Method

Capture what is on the page, not what you wish was there.

The server loads a site in a headless browser, dismisses obvious overlays, records structural and stylistic signals, and segments the page into visual modules. From there it derives a shared grammar that can guide regeneration, review, or design handoff without drifting into invented polish.

Browser pass One context per site, desktop viewport first, then a mobile probe for responsive evidence.
Signal extraction Headings, actions, nav density, section rhythm, imagery weight, forms, and motion cues.
Synthesis Summaries stay aspect-based so hierarchy, color, layout, and interaction remain separately reviewable.
Output

Turn screenshots and signals into a buildable brief.

A Style Trace result does more than summarize visual taste. It tells you what must remain stable, where the evidence is weaker, how modules are sequenced, and which adaptations are safe when you swap in original copy, assets, and product framing.

Design grammar Ten focused aspects with confidence levels and evidence paths.
Style invariants System-level constraints that should survive regeneration.
Review contract Must-match and must-avoid checks for screenshot review.
Originality boundary Reuse hierarchy and spacing lessons, not brand names, protected imagery, or one-to-one composition.
Prompt-ready brief Enough guidance to direct downstream generation without collapsing into category clichés.

The useful part is not “make it feel modern.” The useful part is knowing which layout, chrome, rhythm, and constraint signals are carrying the identity.

Built for reviewable regeneration, not decorative imitation.
Typical flow
1. Trace a reference Analyze one or more public pages and collect evidence in a local artifact.
2. Generate from constraints Use the grammar, blueprint, and risks as hard guidance for a new artifact.
3. Compare screenshots Check first-screen hierarchy, module order, CTA clarity, and drift from the reference signals.
